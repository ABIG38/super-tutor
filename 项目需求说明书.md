## 1. 项目背景与目标

### 1.1 背景

当前考研/学业规划市场存在以下痛点：

- **信息过载**：学生需要翻阅大量教材、历年真题、考纲，难以高效整合
- **规划盲目**：缺乏个性化、基于真实教材内容的复习计划生成工具
- **AI 幻觉问题**：通用大模型在回答专业课时经常捏造知识点或题目，且不标注信息来源
- **上下文受限**：多数 AI 产品无法同时处理整本教材（数十万字）及其配套真题

### 1.2 项目目标

构建一款 **无幻觉、超长上下文、带溯源引用** 的学业/考研规划 Agent，具备以下核心能力：

1. **长文本理解**：支持一次性载入多本教材（百万 Token 级别上下文），进行跨章节、跨教材的全局分析
2. **可信溯源**：所有输出必须标注信息来源于哪本教材、哪个章节，杜绝无中生有
3. **智能规划**：基于真实教材内容，自动生成按天/周拆解的个性化复习计划
4. **精准检索**：针对专业课专有名词（算法名称、硬件指令、定理等），同时支持语义搜索与关键词匹配

## 2. 用户角色

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| 考研学生 | 正在备考研究生入学考试的学生 | 上传教材/真题，获取个性化复习计划，做知识点问答 |
| 学业规划师 | 辅导机构老师或学长学姐 | 基于特定教材为学生批量生成规划，审核溯源内容 |
| 教务管理者 | 院系或辅导机构管理人员 | 查看学生使用情况，管理公共知识库 |

**阶段一重点覆盖「考研学生」角色。**

## 4. 非功能需求

### 4.1 性能要求

| 编号 | 要求 | 指标 |
|------|------|------|
| NF-01 | 文档处理速度 | 100 页 PDF 解析时间 ≤ 30 秒 |
| NF-02 | 问答响应时间 | 首次问答 ≤ 5 秒，缓存命中后 ≤ 2 秒 |
| NF-03 | 上下文容量 | 支持 ≥ 1M Token 的上下文窗口 |
| NF-04 | 并发支持 | 支持 10+ 用户同时使用 |

### 4.2 准确性要求

| 编号 | 要求 | 指标 |
|------|------|------|
| NF-05 | 溯源准确率 | ≥ 98% 的回答包含正确来源标注 |
| NF-06 | 幻觉率 | ≤ 2%（通过人工抽样评估） |
| NF-07 | 专业术语匹配 | BM25 检索命中率 ≥ 90% |

### 4.3 可用性要求

| 编号 | 要求 | 指标 |
|------|------|------|
| NF-08 | 系统可用性 | 核心服务 99.5% 可用 |
| NF-09 | 易用性 | 非技术用户可在 5 分钟内完成上传 + 首次问答 |

### 4.4 安全与隐私

| 编号 | 要求 | 描述 |
|------|------|------|
| NF-10 | 数据本地化 | 向量数据库及用户数据存储在本地文件系统 |
| NF-11 | 文档权限 | 用户只能访问自己上传的文档 |
| NF-12 | API Key 安全 | 云端 API Key 通过环境变量配置，不硬编码在代码中 |

## 5. 系统架构

### 5.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户界面 (CLI / Web)                      │
└──────────────────────────┬──────────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────────┐
│                      Orchestrator (Agent)                        │
│                   LangChain Agent 编排层                          │
│     (路由到文档处理 / 检索 / 规划 等工具)                          │
└────┬──────────┬──────────────┬──────────────────┬───────────────┘
     │          │              │                  │
┌────▼───┐ ┌───▼──────┐ ┌─────▼──────┐  ┌───────▼────────┐
│  文档    │ │  检索     │ │  规划       │  │  对话管理        │
│  引擎    │ │  引擎     │ │  引擎       │  │  (上下文/记忆)    │
└────┬───┘ └───┬──────┘ └─────┬──────┘  └───────┬────────┘
     │          │              │                  │
┌────▼──────────▼──────────────▼──────────────────▼─────────────┐
│                       基础设施层                                 │
│  ┌──────────┐  ┌──────────────┐  ┌─────────────────────────┐   │
│  │ ChromaDB │  │ 文件系统      │  │ LLM API (超长上下文)     │   │
│  │ FAISS    │  │ (原始文档)    │  │ Context Caching         │   │
│  └──────────┘  └──────────────┘  └─────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 模块一：文件处理与超长上下文引擎（Document & Context Engine）

**职责**：负责文档的上传、解析、语义切分，并管理超长上下文的缓存。

**核心流程**：

1. **文档解析**：接收 PDF/Word/MD/TXT 文件，提取完整文本
2. **语义切分**：使用 `RecursiveCharacterTextSplitter` 按自然段落和章节标题切分，而非固定字数
3. **向量化存储**：将切分后的文本块向量化，存入 ChromaDB/FAISS
4. **上下文缓存**：对固定教材启用 LLM API 的 Context Caching，降低延迟和成本

**关键参数**：
- 切分块大小：500-1000 tokens（根据章节边界动态调整）
- 块重叠：100-200 tokens
- 向量模型：text-embedding-3-small 或同级别模型

### 5.3 模块二：无幻觉带溯源的检索系统（RAG with Citation）

**职责**：负责精准检索文档内容，并在生成回答时强制标注来源。

**核心流程**：

```
用户提问
    │
    ▼
┌──────────────┐     ┌──────────────┐
│ 向量检索       │     │ BM25 关键词   │
│ (语义匹配)     │     │ (精确匹配)     │
└──────┬───────┘     └──────┬───────┘
       │                    │
       └────────┬───────────┘
                ▼
       ┌────────────────┐
       │ 结果融合 + 重排序 │
       └───────┬────────┘
                │
                ▼
       ┌──────────────────────────────────┐
       │ LLM 生成 (强制溯源 System Prompt)  │
       │ • 必须标注 [来源文档名]             │
       │ • 不足时声明"通用知识补充"          │
       └──────────────────────────────────┘
```

**System Prompt 核心约束**（在代码中硬编码）：

```
你是一个严谨的学术助手。
1. 请优先使用 <context> 标签内的检索内容回答问题。
2. 在你的回答中，必须用 [来源文档名：章节名] 标注信息来源。
3. 如果 <context> 中的信息不足以回答问题，你可以使用自身知识库补充，
   但必须明确声明："以下内容基于通用知识补充，不在上传文档中：..."
4. 禁止编造来源或引用不存在的文档。
```

### 5.4 模块三：Agent 规划模块（Planning Module）

**职责**：基于教材内容，自动生成个性化的学业/考研复习计划。

**核心流程**：

1. **教材分析**：检索教材的章节目录、前言、重难点标识
2. **时间拆解**：根据用户输入的复习总时长（如"30 天"），将内容分配到各周/天
3. **计划生成**：输出 Markdown 格式的详细计划，包含每天的学习内容、配套习题建议
4. **动态调整**：根据用户反馈的知识点掌握程度，调整后续安排

**规划输出格式示例**：

```markdown
## 第 1 周：数据结构基础（3月1日 - 3月7日）

### 第 1 天：绪论与算法分析
- [来源：《数据结构（C语言版）》第1章]
- 学习内容：数据结构基本概念、算法时间复杂度分析
- 配套习题：教材课后习题 1.1 - 1.5
- 预计用时：2.5 小时

### 第 2 天：线性表
- [来源：《数据结构（C语言版）》第2章]
- 学习内容：顺序表与链表的实现与对比
- 配套习题：教材课后习题 2.1 - 2.8
- 预计用时：3 小时
```

## 6. 技术栈选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 开发环境 | Cursor / Windsurf | AI IDE，自然语言生成代码，工程级理解 |
| 核心框架 | Python + LangChain | Agent 编排、记忆管理、工具调用、RAG 数据流 |
| LLM API | 支持 1M+ Token 窗口的云端大模型 | 塞入整本教材，超长上下文分析，Context Caching |
| 向量数据库 | ChromaDB / FAISS | 轻量级，本地文件存储，无需独立数据库服务器 |
| 文档解析 | PyMuPDF / python-docx / markdown-it | PDF/Word/MD 解析 |
| 文本切分 | LangChain RecursiveCharacterTextSplitter | 按语义边界切分，保留知识点完整性 |
| 关键词检索 | BM25 (rank_bm25) | 专业术语精确匹配，弥补向量检索的不足 |
| 前端（可选） | Streamlit / Gradio | 快速搭建 Web 交互界面 |

## 7. 数据流设计

### 7.1 文档上传与索引流程

```
用户上传 PDF/Word
        │
        ▼
   文档解析（提取文本 + 保留章节结构）
        │
        ▼
   语义切分（按章节/段落切分）
        │
        ▼
   向量化（Embedding） ───► ChromaDB（向量索引）
        │
        ▼
   BM25 索引构建 ───► 本地索引文件
        │
        ▼
   上下文缓存预热 ───► LLM API Cache（可选）
        │
        ▼
   完成，可开始问答
```

### 7.2 问答流程

```
用户提问
    │
    ▼
   Agent 判断意图（问答 / 规划 / 闲聊）
    │
    ▼
   若为问答 → 执行混合检索
        ├── 向量检索（Top-K 语义相似文档块）
        └── BM25 检索（Top-K 关键词匹配文档块）
    │
    ▼
   结果融合 + 去重 + 重排序
    │
    ▼
   构建 Prompt（System Prompt + 检索结果 + 用户问题）
    │
    ▼
   调用 LLM API（命中缓存则极速返回）
    │
    ▼
   解析输出（提取引用标记）
    │
    ▼
   返回带溯源的回答
```

### 7.3 规划生成流程

```
用户指令："制定 30 天复习计划"
        │
        ▼
   Agent 识别 → 调用 Planning Tool
        │
        ▼
   检索教材章节目录 + 前言 + 重难点
        │
        ▼
   计算时间分配（30天 ÷ 章节数）
        │
        ▼
   逐天生成计划内容（含来源标记）
        │
        ▼
   输出 Markdown 格式计划
```

## 9. 风险评估

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| LLM API 费用超预期 | 中 | 中 | 优先使用 Context Caching 节省成本；实现本地小模型兜底 |
| 文档解析质量差（含公式/图表） | 高 | 中 | 公式用 LaTeX 表示保留语义；图表暂时跳过或添加文字说明 |
| 检索召回率不足 | 高 | 低 | 混合检索 + 重排序；支持用户反馈"没找到"以触发重新索引 |
| 用户隐私顾虑 | 中 | 低 | 数据存储本地化；明确隐私政策 |
